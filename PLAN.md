# 纯网页端局域网传文件独立项目方案

## Summary
新建一个独立的 Web 项目，不改现有桌面工程。方案改为 `浏览器页面 + 同站点信令服务 + WebRTC 点对点传输`，设备通过 `房间码` 进入同一会话后建立连接，使用浏览器文件/文件夹选择器发送，接收端通过浏览器下载保存。  
这个方案可行，但它不是当前桌面版的等价迁移。必须明确放弃或弱化这些能力：`UDP 局域网自动发现`、`系统托盘/右键菜单`、`自定义接收目录`、`纯本地直连无信令`。保留目标是：文件/文件夹发送、每次确认接收、基础历史、刷新后恢复续传。

## Key Changes
### 1. 项目形态与技术栈
- 新项目采用 `Python Web 服务 + 浏览器前端`，建议同一服务同时提供页面和信令。
- 后端建议固定为 `FastAPI + WebSocket`：
  - `GET /` 提供前端页面
  - `WS /ws/signaling` 处理房间码、offer/answer、ICE candidate、传输控制消息
- 前端建议固定为 `TypeScript SPA`，由同站点静态托管；实现时可选 `React`，但协议与数据模型不要依赖框架特性。
- 传输通道固定为 `WebRTC DataChannel`，不走服务器中转文件内容。
- 历史与续传状态固定存于 `IndexedDB`，不依赖服务器数据库保存文件分块。

### 2. 产品行为重定义
- 设备发现：
  - 删除局域网自动发现。
  - 改为用户手动输入或复制 `房间码`，双方进入同一房间建立连接。
- 发送能力：
  - 支持浏览器文件选择器选文件。
  - 支持 `webkitdirectory` 方式选文件夹，保留相对路径。
  - 支持拖拽文件/文件夹到页面。
- 接收能力：
  - 接收端每次都确认本次传输。
  - 浏览器通过下载流保存，默认落到浏览器下载目录；不提供应用级自定义目录。
- 历史/续传：
  - 保留历史记录。
  - 支持刷新后恢复未完成会话。
  - 恢复范围仅限同一浏览器同源环境；跨浏览器、清缓存后不保证恢复。
- 信任模型：
  - 不做长期设备信任。
  - 每次进入房间后，本次传输单独确认。

### 3. 核心接口与数据流
- 信令消息固定包含：
  - `join_room`: `room_code`, `client_id`, `display_name`
  - `peer_joined` / `peer_left`
  - `webrtc_offer`, `webrtc_answer`, `ice_candidate`
  - `transfer_request`: 会话 ID、文件清单、总大小
  - `transfer_accept` / `transfer_reject`
  - `resume_request` / `resume_state`
- 文件元数据固定包含：
  - `file_id`, `relative_path`, `size`, `sha256`, `chunk_size`, `chunk_count`
- 浏览器端 IndexedDB 至少包含 3 类 store：
  - `sessions`: 会话状态、方向、房间码、总大小、创建/开始/完成时间
  - `files`: 文件清单、相对路径、大小、校验、完成状态
  - `chunks`: 分块完成位图或已完成 chunk 索引
- 传输流程固定为：
  1. 双方进入房间码
  2. 信令服务撮合 WebRTC 连接
  3. 发送端提交 `transfer_request`
  4. 接收端确认
  5. 双方交换 `resume_state`
  6. 发送端仅发送缺失 chunk
  7. 每个文件完成后做 `SHA-256` 校验
  8. 全部完成后写历史并清理临时会话状态
- 刷新恢复固定为：
  - 页面加载时从 IndexedDB 扫描未完成会话
  - 若用户重新进入原房间码，则重新发起 resume 协商
  - 若对端不在线，会话保留为“可恢复”

### 4. 与当前桌面项目的关系
- 不复用当前 `PySide6` UI、托盘、Jump List、资源管理器右键菜单。
- 可复用的只有产品规则和部分传输/状态模型概念，不直接复用现有 TCP/UDP 实现。
- 当前桌面项目继续独立维护；Web 项目单独目录、单独构建、单独分发。
- 不要求与桌面版协议互通，除非后续明确追加“桌面版与网页版互传”。

## Test Plan
- 房间连接：
  - 两个浏览器进入同一房间码可建立连接。
  - 不同房间码互不可见。
  - 一端离开后另一端收到断开状态。
- 传输功能：
  - 单文件发送成功。
  - 文件夹发送成功且保留相对路径。
  - 拖拽发送与文件选择发送结果一致。
  - 中文文件名、空格路径、深层目录正常。
- 接收确认：
  - 接收端拒绝后发送端正确显示失败/拒绝。
  - 每次新会话都需要重新确认。
- 续传：
  - 传输中断后页面刷新，再进入同一房间码可恢复。
  - 只重传缺失 chunk。
  - 文件校验失败时会话标记失败，不写完成态。
- 历史记录：
  - 已完成、失败、待恢复会话在刷新后仍可见。
  - 清缓存后历史丢失，行为符合预期提示。
- 浏览器兼容：
  - 首测至少覆盖 Chrome/Edge 最新版。
  - Firefox/Safari 不作为首版必达，除非后续单独加兼容目标。

## Assumptions
- 这是一个独立新项目，不修改当前桌面项目代码结构。
- 首版目标是“纯网页可用”，不是“完全替代桌面版”。
- 信令服务与前端一起部署，不引入第三方信令平台。
- 首版默认浏览器环境为 Chromium 系浏览器。
- 首版不做服务器中转文件内容，不做账号系统，不做公网 NAT 穿透兜底，不做系统级右键菜单集成。
- 首版不支持局域网自动发现；房间码是唯一连接入口。
